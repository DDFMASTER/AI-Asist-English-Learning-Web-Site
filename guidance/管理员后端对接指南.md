# 管理员后端对接指南

---

## 一、概述

本文档供前端开发人员参考，说明管理员功能涉及的后端 API 端点、请求参数、响应格式，以及 Vue 页面设计建议。

管理员身份由 `user` 表的 `role` 字段控制，取值为 `ENUM('normal','vip','admin')`。用户登录成功后，响应中会携带 `role` 字段，前端据此判断是否展示管理入口。

---

## 二、管理员身份识别

### 登录接口回顾

**`POST /api/user/login`**

请求（form-urlencoded）：

| 参数 | 类型 | 说明 |
|---|---|---|
| username | String | 用户名 |
| password | String | 明文密码 |

成功响应：

```json
{
  "success": true,
  "message": "登录成功",
  "userId": 1,
  "username": "admin01",
  "role": "admin",
  "studyPurpose": "四级",
  "experience": 120,
  "literacy": 3500
}
```

> **关键：** `role` 字段为 `"admin"` 时，前端首页应展示管理入口（例如侧边栏或顶栏增加"管理"菜单项）。

---

## 三、通用约定

### 3.1 管理员鉴权

所有管理员接口均要求传入 `adminUserId` 参数，后端会根据 `user` 表校验其角色是否为 `admin`。

- **鉴权失败响应：**

```json
{
  "success": false,
  "message": "无管理员权限"
}
```

> **前端注意：** 收到此响应时应提示用户并跳转回首页或登录页。

### 3.2 请求方式

| 接口 | 方法 | Content-Type |
|---|---|---|
| 用户列表 | GET | —（Query String） |
| 其余全部 | POST | application/x-www-form-urlencoded |

### 3.3 响应格式

所有接口统一返回 JSON：

```json
{
  "success": true | false,
  "message": "操作结果描述"
}
```

除了用户列表接口会额外携带 `users` 数组。

### 3.4 枚举值约束

**学习阶段 / 难度等级：** `初中` `高中` `四级` `六级` `考研` `托福`

**用户角色：** `normal` `vip` `admin`（仅 set-role 接口传 normal/vip，不能将用户设为 admin）

---

## 四、API 端点详述

### 4.1 用户管理

#### 4.1.1 获取全部用户列表

```
GET /api/admin/user/list?adminUserId=<adminId>
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |

成功响应：

```json
{
  "success": true,
  "users": [
    {
      "userId": 1,
      "username": "admin01",
      "role": "admin",
      "studyPurpose": "四级",
      "literacy": 5000,
      "experience": 200,
      "lastLogin": "2026-06-29 10:30:00",
      "createdAt": "2026-01-15 08:00:00"
    },
    {
      "userId": 2,
      "username": "zhangsan",
      "role": "normal",
      "studyPurpose": "六级",
      "literacy": 3200,
      "experience": 80,
      "lastLogin": "2026-06-28 18:00:00",
      "createdAt": "2026-03-20 14:00:00"
    }
  ]
}
```

**返回字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| userId | Long | 用户唯一 ID |
| username | String | 登录用户名 |
| role | String | 角色：`normal` / `vip` / `admin` |
| studyPurpose | String | 当前学习阶段 |
| literacy | Integer | 词汇量测试得分 (0-20000) |
| experience | Integer | 经验值 |
| lastLogin | String | 最近登录时间 (yyyy-MM-dd HH:mm:ss)，可能为 null |
| createdAt | String | 注册时间 |

> **前端建议：** 列表展示 userId、username、role、studyPurpose、experience、createdAt；lastLogin 和 literacy 可放到详情或 tooltip。role 可用不同颜色的 tag/badge 区分。

---

#### 4.1.2 创建用户

```
POST /api/admin/user/create
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| username | 是 | String | 新用户名，≤16 字符 |
| studyPurpose | 是 | String | 学习阶段（见枚举约束） |

成功响应：

```json
{
  "success": true,
  "message": "用户创建成功"
}
```

可能失败响应：

| message | 原因 |
|---|---|
| 用户名不能为空 | username 缺失或空 |
| 用户名不能超过16个字符 | username 超长 |
| 用户名已存在 | 用户名重复 |
| 请选择学习阶段 | studyPurpose 缺失或空 |
| 无效的学习阶段 | studyPurpose 不在枚举范围内 |

> **重要：** 管理员创建的用户默认密码为 `123456`，默认角色为 `normal`。创建成功后前端应提示"初始密码为 123456"。

---

#### 4.1.3 删除用户

```
POST /api/admin/user/delete
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| userId | 是 | Long | 要删除的目标用户 ID |

成功响应：

```json
{
  "success": true,
  "message": "用户已删除"
}
```

可能失败响应：

| message | 原因 |
|---|---|
| 缺少目标用户ID | userId 缺失或无效 |
| 不能删除自己的账户 | 管理员试图删除自己 |
| 用户不存在 | userId 对应的用户不存在 |

> **前端建议：** 删除前务必弹出二次确认对话框，明确告知"将同时删除该用户的投票记录和学习记录"。管理员无法删除自己。

---

#### 4.1.4 重置用户密码

```
POST /api/admin/user/reset-password
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| userId | 是 | Long | 目标用户 ID |

成功响应：

```json
{
  "success": true,
  "message": "密码已重置为123456"
}
```

> **前端建议：** 操作前弹出确认框，操作成功后明确提示"密码已重置为 123456"。

---

#### 4.1.5 设置用户角色

```
POST /api/admin/user/set-role
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| userId | 是 | Long | 目标用户 ID |
| role | 是 | String | 目标角色：`normal` 或 `vip` |

成功响应：

```json
{
  "success": true,
  "message": "角色设置成功"
}
```

可能失败响应：

| message | 原因 |
|---|---|
| 无效的角色，仅支持 normal 或 vip | role 不是 normal/vip |
| 不能修改管理员角色 | 目标用户本身是 admin |
| 用户不存在 | userId 对应不存在 |

> **前端建议：** 使用下拉选择框或 radio 切换，选项为"普通用户 (normal)"和"VIP用户 (vip)"。管理员角色不可被修改，对 admin 用户应禁用此操作。

---

### 4.2 文章管理

#### 4.2.1 创建文章

```
POST /api/admin/article/create
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| title | 是 | String | 文章标题 |
| content | 是 | String | 文章正文（英文），MEDIUMTEXT，可容纳约 16MB |
| source | 否 | String | 文章来源，选填 |
| difficulty | 是 | String | 难度等级（见枚举约束） |

成功响应：

```json
{
  "success": true,
  "message": "文章创建成功"
}
```

可能失败响应：

| message | 原因 |
|---|---|
| 文章标题不能为空 | title 缺失或空 |
| 文章内容不能为空 | content 缺失或空 |
| 请选择难度等级 | difficulty 缺失或空 |
| 无效的难度等级 | difficulty 不在枚举范围内 |

> **前端建议：** title 用单行输入框，source 用单行输入框（placeholder 提示选填），difficulty 用下拉选择框，content 用多行文本框（textarea），建议 rows 设大一些（≥15 行）。

---

#### 4.2.2 删除文章

```
POST /api/admin/article/delete
```

| 参数 | 必须 | 类型 | 说明 |
|---|---|---|---|
| adminUserId | 是 | Long | 管理员用户 ID |
| articleId | 是 | Long | 要删除的文章 ID |

成功响应：

```json
{
  "success": true,
  "message": "文章已删除"
}
```

可能失败响应：

| message | 原因 |
|---|---|
| 缺少文章ID | articleId 缺失或无效 |
| 文章不存在 | articleId 对应的文章不存在 |

> **前端建议：** 删除前弹出确认框，告知"将同时删除该文章关联的词汇题、阅读理解题和学习记录"。建议在现有的文章列表页（复用 `/api/article/list`）中为每篇文章增加删除按钮。

## 五、前端页面设计建议

### 5.1 权限入口

登录后根据 `role` 字段判断：

```javascript
// 登录成功后存储用户信息
const user = loginResponse;
localStorage.setItem('userId', user.userId);
localStorage.setItem('role', user.role);

// 路由守卫或模板中
if (user.role === 'admin') {
  // 显示"管理中心"入口
}
```

### 5.2 管理页面路由建议

```
/admin               → 管理首页（仪表盘，可选）
/admin/users         → 用户管理页（列表 + 操作按钮）
/admin/articles      → 文章管理页（列表 + 新建 + 删除）
```

### 5.3 用户管理页 (`/admin/users`)

**顶部操作栏：**
- "新建用户"按钮 → 弹出对话框：输入 username、选择 studyPurpose

**用户列表表格：**

| 列 | 说明 |
|---|---|
| ID | userId |
| 用户名 | username |
| 角色 | role（彩色 tag：admin=红色, vip=金色, normal=灰色） |
| 学习阶段 | studyPurpose |
| 经验值 | experience |
| 注册时间 | createdAt |

**每行操作按钮：**
- 设为 VIP / 设为普通 → 调用 set-role
- 重置密码 → 确认后调用 reset-password
- 删除 → 二次确认后调用 delete（管理员自己不可删）

### 5.4 文章管理页 (`/admin/articles`)

**顶部操作栏：**
- "新建文章"按钮 → 弹出对话框或跳转新页面

**新建文章表单（对话框或独立页面）：**

| 字段 | 控件 | 必填 |
|---|---|---|
| 标题 | `<input>` | 是 |
| 来源 | `<input>` | 否 |
| 难度 | `<select>` | 是 |
| 正文 | `<textarea>`（建议 ≥15 行） | 是 |

**文章列表：**
- 复用 `/api/article/list` 接口获取现有文章
- 每行增加"删除"按钮

### 5.5 交互规范建议

1. **危险操作确认：** 删除用户、删除文章、重置密码都应有二次确认弹窗
2. **操作反馈：** 成功/失败使用 toast 或 message 组件提示
3. **列表刷新：** CUD 操作成功后自动刷新列表
4. **错误处理：** 鉴权失败（"无管理员权限"）时，清除本地用户状态并跳转到登录页
5. **参数校验：** 前端应先做基本校验（非空、长度限制），减少无效请求

## 六、API 速查表

| 功能 | 方法 | URL | 关键参数 |
|---|---|---|---|
| 用户列表 | GET | `/api/admin/user/list` | adminUserId |
| 创建用户 | POST | `/api/admin/user/create` | adminUserId, username, studyPurpose |
| 删除用户 | POST | `/api/admin/user/delete` | adminUserId, userId |
| 重置密码 | POST | `/api/admin/user/reset-password` | adminUserId, userId |
| 设置角色 | POST | `/api/admin/user/set-role` | adminUserId, userId, role |
| 创建文章 | POST | `/api/admin/article/create` | adminUserId, title, content, source?, difficulty |
| 删除文章 | POST | `/api/admin/article/delete` | adminUserId, articleId |

---

## 七、注意事项

- 所有 POST 请求使用 `application/x-www-form-urlencoded`，不是 JSON body
- `adminUserId` 在每次请求中必传，后端无 session/cookie 机制
- 管理员权限在每次请求时实时查询数据库校验，而非缓存
- 删除用户是**物理删除**，会级联清除投票记录和学习记录，不可恢复
- 删除文章是**物理删除**，会级联清除关联的题库数据，不可恢复
- 管理员不能通过 set-role 接口修改其他管理员的角色，也不能删除自己的账户
- 初始密码 `123456` 为明文，前端应在创建/重置成功后提示用户尽快修改密码
